## Change Tested

Tested experiment 2 from the repair matrix: when a live 7DTE SPXW iron condor
reaches short-leg delta `>= 0.20` while still in the `6-7` DTE bucket, the
strategy closes the condor outright instead of holding or re-centering.

## Method

Ran the standalone project `qc/spxw_7dte_exp02_close_high_dte` in QuantConnect
cloud and parsed the saved results payload from
`research_log/results/spxw_7dte_exp02_close_high_dte_20260330.json`. The
project used minute SPX and SPXW data, baseline entry constants, and the same
`IronCondorFinder` as the no-constraints baseline.

## Interpretation

This version was materially worse than `exp01` on aggregate performance. The
hard-close rule did not just give up upside; it still produced a large drawdown
and finished with a negative net return. That does not prove it failed on every
left-tail measure, but it is a weak result for the idea that high-DTE closing
is a robust repair.

## Artifacts And Caveats

- Suspicious artifacts: none obvious from the aggregate QC stats alone.
- Data issues: the saved QC payload reports `backtest_end` as `2025-12-30`, not the intended `2025-12-31`.
- Execution anomalies: the run used pre-refactor code version `c1ae31d`; the later shared `IronCondorRepairManager` cleanup was not part of this tested state.
